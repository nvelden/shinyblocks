# ADR 0018: Custom Next.js Static Documentation Site

## Status

Accepted (2026-05-19)

## Context

`shinyblocks` is a premium, modern Shiny dashboard library inspired by the visual language of `shadcn/ui`. For an aesthetic-driven component system, documentation is not just supporting text—it is a critical extension of the product. The traditional R documentation tool (`pkgdown`) is heavily tied to Bootstrap aesthetics and does not match the modern design tokens, layout flexibility, and premium micro-interactions of the `shinyblocks` brand.

Furthermore, we require high-fidelity component previews, static R-code recipe playgrounds, and versioned changelog layouts that feel native to `ui.shadcn.com`. Adopting a custom documentation site allows us to curate a premium marketing and technical learning surface.

## Decision

Decommission `pkgdown` completely and adopt a custom Next.js 16 + Tailwind CSS v4 static site under the directory `docs-site/`, deployed automatically to GitHub Pages.

### Architectural Rules

1. **Static Export**: The documentation site must be statically exported (`output: "export"`) so it can be served with zero-overhead on standard CDNs and GitHub Pages at the path `/shinyblocks`.
2. **Static R Prerender Pipeline**: Previews on the homepage and within component playgrounds must avoid the heavy weight and load-times of live WASM engines (like Shinylive) in `v1`. Previews are instead generated by a prebuild R script (`scripts/generate-previews.R`) that:
   - Evaluates real `block_*()` R code snippets.
   - Compiles them into static HTML fragments using `htmltools::renderTags()`.
   - Embeds both the rendered HTML and source R code directly into a single unified `preview-manifest.json` at build time.
3. **High-Fidelity Client Runtime Hydration**: To ensure previews reflect active, fully rounded, colored, and interactive shancdn/ui styling:
   - The Next.js layout reads the current `shinyblocks` package runtime assets (`shinyblocks-runtime.js` and `shinyblocks-runtime.css`) and copies them to the public build path.
   - The site dynamically hydrates the static HTML previews client-side via the standard runtime mount lifecycle.
4. **NEWS.md to Changelog Compiler**: To avoid duplication, the changelog section must be automatically generated from the root `NEWS.md` file via a Node/TypeScript prebuild script (`scripts/generate-changelog.ts`), exposing version headers as anchor deep-links.
5. **No Fumadocs/Shinylive in v1**: Fumadocs is dropped in `v1` to keep the codebase simple and lightweight. Shinylive and dynamic code runners are explicitly deferred to `v2`.
6. **Continuous Quality Gates**: Every documentation release must pass standard Next.js compilation, `devtools::check()` package compliance, and Playwright E2E browser tests across Chromium and WebKit.

## Consequences

**Positive:**
- A premium, state-of-the-art documentation surface that aligns 1:1 with modern design conventions.
- Interactive, high-fidelity React-rendered previews of real Shiny dashboard elements.
- Single source of truth for the changelog (`NEWS.md`) and component recipes.
- Instant page transition times and excellent search engine optimization (SEO).

**Negative / Accepted Costs:**
- Maintainers must now run a dual-stack pipeline (R package + Node/Next.js site build steps).
- Example code must be registered both in the showcase R files and the static docs site registry.
- Component API tables require maintained JSON stubs under `content/api/`.

## Verification

Compliance is verified through:
- Automated Playwright E2E test suites covering landing layout, index searching, detail routes, and changelog anchor deep-linking.
- The package's documentation-sync test (`test-doc-coverage.R`) checking reference groups.
- Visual inspection under both light and dark modes at responsive viewport scales (desktop down to 375px mobile).
